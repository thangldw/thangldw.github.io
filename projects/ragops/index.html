The deployment gap

Teams can build an impressive RAG or agent prototype in days. The harder question arrives before production: is this version safe and useful enough to release?

Evidence is usually split across notebooks, vendor traces, prompt logs, and manual reviews. A quality score may improve while latency doubles. An average may hide a critical permission leak. A demo can work while nobody can explain what changed from the accepted baseline.

I reframed the problem as a release contract: the same scenario, the same policies, a recorded baseline, a candidate build, and an explainable go/no-go result that can run locally or in CI.

Product contract

  • Offline by default
  • Deterministic before model-judged
  • Portable scenarios and traces
  • Critical findings override averages
  • Open core; hosted layer optional

Rejected: another RAG app

A reference chatbot would show retrieval and generation, but it would not solve the reusable production problem shared by different AI workflows.

Chosen: evaluation infrastructure

The product evaluates systems rather than owning their model or vector database. That keeps the core horizontal, portable, and commercially extensible.

What this demonstrates

Start from the decision

The product is organized around a go/no-go decision, not around fashionable model components.

Make quality operational

A metric becomes useful when it is versioned, explainable, and able to block a bad release in CI.

Design for handover

Portable contracts, zero-dependency core, ADRs, tests, and explicit limitations make the system easier for another team to own.

Current status

v1.4.0 is public and ready for OSS feedback, FDE interviews, and customer-reviewed offline pilots. The hosted control-plane layer remains an explicitly non-production alpha.
